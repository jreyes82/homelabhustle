Adding Portainer and Migrating My Docker Services

Self-Hosted March 2026 8 min read

I had been running Docker Desktop on Windows 11 for a while — managing everything through the CLI and the basic Docker Desktop UI. It worked, but as the number of containers grew it became harder to keep track of what was running, what ports were mapped where, and what needed updating. I decided to add Portainer on top of my existing Docker setup and migrate all my services into it. Here's how that went.

Why Portainer

Portainer gives you a proper web UI for managing Docker — containers, images, volumes, networks, and stacks all in one place. The key thing for me was stacks — Portainer lets you deploy services using docker-compose files directly from the UI, which makes it much easier to manage multi-container setups and keep everything documented.

I was already using docker-compose files for most of my services, so the migration path was straightforward in theory. In practice, it took a few hours and hit a couple of snags.

Installing Portainer

Since Docker was already running, adding Portainer was just a matter of pulling the image and running it as a container. I ran this in PowerShell:

docker run -d -p 9000:9000 --name portainer `
  --restart=always `
  -v /var/run/docker.sock:/var/run/docker.sock `
  -v portainer_data:/data `
  portainer/portainer-ce:latest

After that, Portainer was available at localhost:9000. First-time setup asks you to create an admin account and connect to the local Docker environment — straightforward.

Tip: The portainer_data volume is where all your Portainer config lives. Never delete it. If you ever need to update Portainer, stop and remove the container but leave the volume alone and your config will survive.

Migrating the Services

I had five services to bring over: the FLAC music bot, a Discord music bot, Aurral, Framerr, and Apache Guacamole. My approach was to take the docker-compose files I already had and deploy them as Portainer stacks one by one.

The process for each service

  • Stop and remove the existing container in Docker Desktop
  • Go to Portainer — Stacks — Add Stack
  • Paste in the docker-compose content
  • Deploy and verify it came up healthy

For most services this worked on the first try. The docker-compose files I already had were clean and Portainer just picked them up without issues. Volumes carried over since they were named volumes — the data was still there.

Where things got complicated

Port conflicts were the main headache. When I had containers still running in Docker Desktop and tried to bring up a Portainer stack pointing to the same ports, Docker would throw a bind error. The fix was making sure the old container was fully stopped and removed before deploying the stack — obvious in hindsight, but easy to miss when moving through multiple services quickly.

Guacamole was the most involved migration. It runs three containers — guacd, the Guacamole web app, and a database — so the compose file is more complex. Once I had the compose file right it came up cleanly.

The Result

After everything was migrated, the difference in visibility was immediately noticeable. Instead of running docker ps to check what's running, I can see all my stacks in one place, check container health, view logs, and manage updates — all from the browser.

What I'd Do Differently

Stop all existing containers before starting any migrations rather than doing them one at a time. The port conflicts I hit were a direct result of having old containers still running while trying to bring up new stacks. A clean stop-everything-first approach would have saved some troubleshooting time.
