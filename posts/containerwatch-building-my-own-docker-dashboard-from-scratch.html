ContainerWatch: Building My Own Docker Dashboard from Scratch

Homelab April 2026 8 min read

I ran Portainer for a long time. It works, it's stable, and for most people it's all they need. But over time I kept running into the same friction — it's a heavy application built for teams managing large environments, and I'm one person managing a homelab. I wanted something lighter, faster, and built specifically around what I actually do every day. So I built ContainerWatch.

Why Not Just Use Portainer

Portainer does a lot. That's the problem. Half the features I never touch, and the ones I care about most — knowing whether my container images are out of date — require navigating through multiple screens and still don't give me a clean at-a-glance answer. I also wanted Discord notifications when something needed attention, and I wanted to be able to scan Python dependencies inside my containers without SSH-ing in manually.

None of that was impossible with Portainer, but none of it was easy either. The simpler option was to build exactly what I wanted.

What ContainerWatch Does

ContainerWatch is a Flask and Python web application that runs as a Docker container and communicates with the Docker socket to monitor everything else running on the host. The core features are:

  • Image update checking — compares the digest of each running container's image against the remote registry. Supports Docker Hub, GHCR, and LSCR. Shows clearly which containers are up to date and which have updates available.
  • One-click updates — pull the latest image and restart a container directly from the dashboard with a live log stream showing exactly what's happening.
  • Dependency scanning — runs pip inside Python-based containers to check for outdated packages, and can upgrade them in place with a live output stream.
  • Live container logs — tail logs from any running container directly in the browser without touching the CLI.
  • Discord notifications — sends a webhook notification when a scan finds outdated containers, with a configurable auto-scan interval.
  • Scan history — stores every scan result in SQLite so you can see how your containers have changed over time.

The Stack

The backend is Python and Flask. It connects to the Docker socket via the Docker SDK, which gives it full access to inspect containers, pull images, stream logs, and execute commands inside running containers. All scan history and settings are stored in SQLite via a mounted volume so data persists across container restarts.

The frontend is a single HTML file served by Flask. No JavaScript frameworks — just vanilla JS making API calls to the Flask backend. It keeps the whole thing simple and fast.

It runs behind Cloudflare Zero Trust so it's accessible remotely without exposing any ports to the public internet.

services:
  containerwatch:
    build: .
    container_name: containerwatch
    ports:
      - "6080:5000"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - containerwatch-data:/data
    restart: unless-stopped
    environment:
      - GITHUB_TOKEN=${GITHUB_TOKEN:-}
      - DB_PATH=/data/containerwatch.db

volumes:
  containerwatch-data:

The Hard Part - Registry Checks

The trickiest part of the build was getting image update checks to work reliably across different registries. The approach is to compare the digest of the locally running image against the digest of the latest tag on the remote registry. If they match, the container is up to date. If they don't, there is an update available.

The complication is that every registry handles authentication differently. Docker Hub uses one token endpoint, GHCR uses another, and LSCR (LinuxServer) uses a third. Each one also has slightly different behavior around manifest formats and the headers required to get a proper digest back.

Getting all three working consistently — and handling edge cases like private images, missing tags, and registry timeouts — took more iteration than any other part of the project. The final implementation uses a shared requests session with connection pooling and retries, which also helped with performance when scanning a large number of containers at once.

Note: The registry check compares image digests, not version tags. This means it catches updates even when the tag stays the same (like latest), which is the more reliable approach.

What I Use It For Daily

The features I reach for most are image update checking and Discord notifications. I have ContainerWatch set to run an automatic scan on a schedule, and when something is outdated it pings my Discord server. I open the dashboard, see what needs updating, click update, watch the log stream, done. The whole process takes about two minutes per container and I never have to touch the CLI unless something goes wrong.

The dependency scanner gets used less frequently but it's useful for my Python-based containers — particularly the bots and custom services where I want to keep packages current without rebuilding the image.

What I Learned

Building ContainerWatch taught me more about how Docker actually works under the hood than years of just using it did. Working directly with the Docker SDK, understanding image manifests and digest comparison, handling the Docker socket from inside a container — all of that gave me a much deeper understanding of the infrastructure I run every day.

It also reinforced something I already believed: the best tool for a specific job is often a small one built exactly for that job. ContainerWatch does less than Portainer and that is entirely the point.
